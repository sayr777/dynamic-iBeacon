### 🔹 1. Стратегическое позиционирование
- **Цель сервиса:** Предоставить партнёрам доверенный API для уточнения координат ТС в зонах, где GPS/ГЛОНАСС недостаточно точен (городские каньоны, тоннели, зоны глушения), используя уже развёрнутую сеть iBeacon (BLE метки) на остановках общественного транспорта.
- **Уникальное преимущество:** Распознавание остановки выполняется без раскрытия статических идентификаторов в эфире. Дешифровка и маппинг на координаты происходят в защищённом облачном контуре РНИС.
- **Бизнес-логика:** Инфраструктура уже оплачена и работает. Сервис монетизируется как B2B-подписка с тарификацией по объёму запросов и SLA, что соответствует модели «платный доступ доверенных интеграторов» из презентации.

---
### 🔹 2. Архитектура облачного сервиса
| Компонент | Назначение |
|-----------|------------|
| **API Gateway** | Аутентификация партнёров, rate-limiting, маршрутизация, валидация схемы запроса |
| **Ingestion Service** | Приём payloads `(timestamp, major, minor, rssi, uuid, radioMAC)`, проверка временных окон, дедупликация |
| **Decryption & Mapping Engine** | Дешифровка `major/minor` через 128-битный региональный ключ (AES-128), сопоставление с реестром остановок, возврат точных координат и метаданных |
| **Fusion Module (опционально)** | Взвешенное объединение BLE-координат с GPS партнёра на основе RSSI и заявленной точности GPS |
| **Key Management & Config Store** | KMS/HSM для хранения ключей, ротация по расписанию, версияция справочника остановок |
| **Monitoring & Audit** | Метрики latency, error rate, аномалии сигнала, логи доступа для compliance |

**Технологический стек (рекомендация):** 
- Контейнеризация: Kubernetes / EKS/GKE
- БД: PostgreSQL (реестр остановок, метаданные), Redis (кэш маппинга ключ-остановка)
- Очереди: Kafka/RabbitMQ (для асинхронной аналитики и биллинга)
- Инфраструктура: Terraform + CI/CD, OpenTelemetry для трейсинга

---
### 🔹 3. Модель безопасности и доверенного доступа
| Угроза | Реакция системы |
|--------|-----------------|
| **Считывание эфира** | Партнёр видит только динамические `major/minor`. Постоянный ID остановки не передаётся в эфир и не возвращается в API. |
| **Подбор ключа** | 128-битный ключ хранится только в KMS, не передаётся по радио. Перебор криптографически нецелесообразен. |
| **Подмена точки / спуфинг** | Валидация `radioMAC` против инвентарной базы, проверка физически невозможных перемещений, отклонение запросов с аномальным RSSI. |
| **Replay-атаки** | Строгая проверка `timestamp` (окно ≤ 3 сек), nonce-валидация на уровне Gateway. |
| **Утечка данных** | Минимизация ответа: только уточнённые координаты, `stop_id`, `confidence`, временная метка. Логирование без PII. |

### ⚖️ Ключевые юридические пункты для включения в договор
- **Обработка данных**: Партнёр передаёт только технические метаданные (BLE-параметры, временные метки, MAC-адреса). Персональные данные не обрабатываются.
- **Ответственность**: Ограничение ответственности поставщика — суммой платежей за последний расчётный период.
- **Аудит**: Поставщик обязуется предоставлять отчёт о соответствии требованиям безопасности по запросу партнёра (не чаще 1 раза в год).
- **Ротация ключей**: Региональный 128-битный ключ может быть заменён по расписанию или при инциденте. Партнёр обязуется поддерживать актуальность SDK.
- **Прекращение доступа**: При нарушении условий использования (спуфинг, превышение rate limit, попытка реверс-инжиниринга) доступ приостанавливается в одностороннем порядке.

### 📱 Пример интеграции для Android (Kotlin)

```kotlin
// LocationRefineClient.kt
class LocationRefineClient(
    private val apiKey: String,
    private val baseUrl: String = "https://api.t1-location.ru"
) {
    private val httpClient = OkHttpClient.Builder()
        .addInterceptor { chain ->
            val request = chain.request().newBuilder()
                .addHeader("X-API-Key", apiKey)
                .addHeader("Content-Type", "application/json")
                .build()
            chain.proceed(request)
        }
        .build()

    suspend fun refineLocation(payload: RefineRequest): RefineResponse? {
        val json = Json.encodeToString(payload)
        val request = Request.Builder()
            .url("$baseUrl/v1/location/refine")
            .post(json.toRequestBody("application/json".toMediaType()))
            .build()
        
        return try {
            val response = httpClient.newCall(request).execute()
            if (response.isSuccessful) {
                Json.decodeFromString(response.body?.string() ?: "")
            } else null
        } catch (e: Exception) {
            Log.e("LocationRefine", "Request failed", e)
            null
        }
    }
}
```

## 4️⃣ Согласование с РНИС: план изоляции микросервисов

### 🔐 Цели изоляции
- Независимое масштабирование сервиса для партнёров без влияния на ядро РНИС
- Чёткое разделение зон ответственности: дешифровка и реестр — критичные компоненты, вынесенные в отдельный security domain
- Возможность ротации ключей и обновления справочника без остановки основного контура

### 🧩 Архитектурная схема взаимодействия

```
┌─────────────────────┐     ┌─────────────────────┐
│   Партнёрский API   │     │   Ядро РНИС         │
│   (новый контур)    │     │   (существующий)    │
└────────┬────────────┘     └────────┬────────────┘
         │                           │
         │   ┌─────────────────┐   │
         └──►│  Shared Services│◄──┘
             │  • KMS/HSM      │
             │  • Registry DB  │  ← реплика "только чтение"
             │  • Audit Log    │
             └─────────────────┘
```

### 📋 План работ по изоляции

| Этап | Задача | Ответственный | Срок |
|------|--------|---------------|------|
| 1 | Выделить микросервис `decryption-service` с чётким контрактом (gRPC/REST) | Архитектор РНИС | Неделя 1 |
| 2 | Настроить репликацию справочника остановок в `registry-read-replica` | DevOps | Неделя 2 |
| 3 | Интегрировать KMS (HashiCorp Vault / Yandex Cloud KMS) для хранения 128-битного ключа | Безопасность | Неделя 2-3 |
| 4 | Реализовать health-check и circuit breaker между сервисами | Backend-команда | Неделя 3 |
| 5 | Провести нагрузочное тестирование изолированного контура | QA / Performance | Неделя 4 |
| 6 | Зафиксировать контрактные интерфейсы в репозитории `t1-api-contracts` | Технический писатель | Неделя 4 |

### 🔐 Требования к безопасности изолированных компонентов
- **KMS**: Ключ шифрования никогда не покидает защищённый модуль. Дешифровка выполняется внутри доверенного исполнения (TEE, если доступно).
- **Доступ к реестру**: Только `SELECT` через read-only пользователя. Изменения — только через основной контур РНИС с репликацией.
- **Аудит**: Все вызовы `decryption-service` логируются с указанием `partner_id`, `timestamp`, `stop_id` (без major/minor в явном виде).
- **Сеть**: Микросервисы размещаются в отдельной подсети с правилами firewall, разрешающими только необходимые порты от API Gateway.

### 🔹 1. Обновлённая формулировка для юридического отдела
> **Обработка данных:** Партнёр передаёт исключительно технические метаданные (BLE-параметры, временные метки, MAC-адреса радиомодуля) и учётный идентификатор приложения (`client_id`). Указанный идентификатор является техническим маркером интеграции, не содержит сведений о физических лицах, водителях или пассажирах, и используется исключительно для биллинга, контроля квот и аудита доступа. Персональные данные в рамках сервиса не обрабатываются.

### 🔹 2. Варианты передачи `client_id` в API

| Способ | Преимущества | Рекомендация |
|--------|--------------|--------------|
| **Заголовок `X-Client-Id`** | Читается на уровне API Gateway до парсинга тела, удобен для rate-limiting и роутинга | ✅ **Основной вариант** |
| Поле `client_id` в JSON-теле | Видно в payload, удобно для отладки | ⚠️ Дублировать в заголовке для безопасности |
| Вывод из `X-API-Key` | Минимальные изменения в клиенте | ❌ Не подходит, если у партнёра несколько приложений на одном аккаунте |

**Рекомендуемый формат:** `X-Client-Id: UUID v4` или партнёрский строковый идентификатор `^[A-Za-z0-9_\-]{8,32}$`

### 🔹 4. Интеграция с биллинговой подсистемой

| Компонент | Роль `client_id` |
|-----------|------------------|
| **API Gateway** | Rate-limiting, квотирование, маршрутизация на основе `client_id` + `X-API-Key` |
| **Metering Service** | Потребляет Kafka-топик `request.events`, агрегирует счётчики по `client_id` с granularity 1 мин |
| **Billing DB** | Хранит `client_id → partner_account → tariff_plan → usage_period` |
| **Invoice Generator** | Формирует ежемесячный счёт с разбивкой: базовый пакет + overage по каждому `client_id` |
| **Dashboard** | Партнёр видит статистику по приложениям, может отключать/перевыпускать `client_id` без смены API-ключа |

**Схема потока данных:**
```
Client → API Gateway (валидация X-Client-Id) → Ingestion → Kafka(request.events)
Kafka → Metering Service (агрегация 1m/1h) → Billing DB
Billing DB → Invoice Scheduler (1-е число месяца) → Partner Portal
```

| **БД (чтение справочника)** | 0,0002 ₽ | Реплика "только чтение", индексы по stop_id |
| **Сеть (вход/выход трафика)** | 0,0001 ₽ | ~500 байт запрос + ~300 байт ответ |
| **Мониторинг + логирование** | 0,0002 ₽ | Prometheus metrics + Loki log entry |

| **Итого переменные затраты** | **≈ 0,0028 ₽/запрос** | ~0,28 копейки |

### 🔹 Постоянные затраты (распределяются на объём)

| Показатель | Значение |
|------------|----------|

# Чистый денежный поток по месяцам (строка 20, столбцы C:N)

=IRR(C20:N20)

=MATCH(TRUE, CUMULATIVE_CF!C2:N2>0, 0)

### 2.1. Показатели качества услуг

| Показатель | Значение | Метод измерения | Период контроля |

| Уровень | Описание | Пример | Целевое время реакции |

| **P4 (Низкий)** | Косметические дефекты, запросы на изменение | Опечатка в документации | ≤ 2 рабочих дня |

### 4.1. Состав обрабатываемых данных

```
✅ Разрешено к передаче:
   • timestamp (время события, UTC)
   • uuid, major, minor (BLE-параметры метки)
   • rssi (уровень сигнала)
   • radio_mac (MAC-адрес передатчика)
   • client_id (учётный идентификатор приложения)
   • опционально: gps_lat, gps_lon, gps_accuracy_m

❌ Запрещено к передаче:
   • ФИО водителя / пассажира
   • Номер телефона, email
   • Государственный регистрационный знак ТС
   • Идентификаторы аккаунтов в сторонних сервисах
   • Любые данные, позволяющие идентифицировать физическое лицо
```

### 4.2. Цели обработки

1. Доверенная дешифровка идентификатора остановочного пункта
2. Возврат уточнённых геокоординат для повышения точности навигации
3. Учёт объёма потреблённых запросов для биллинга
4. Мониторинг качества оказания услуг и выявление аномалий
5. Формирование агрегированной статистики для отчётности Заказчику

### 4.3. Гарантии защиты данных

- Передача данных осуществляется только по защищённому каналу TLS 1.2+
- Хранение 128-битного ключа дешифровки — в аппаратном модуле KMS/HSM
- Логирование: в логах фиксируется SHA-256 хеш `client_id`, но не полный идентификатор
- Срок хранения сырых запросов: ≤ 72 часов (для отладки)
- Срок хранения агрегированных метрик: ≤ 12 месяцев
- Регулярный аудит безопасности: не реже 1 раза в год

## 1️⃣ Обоснование общественной значимости проекта

### 🔹 Проблема
| Аспект | Текущая ситуация | Последствия |
|--------|-----------------|-------------|
| **Точность фиксации остановок** | Погрешность GPS 10–30 м в городской застройке | Ошибки в субсидировании перевозчиков, некорректная статистика пассажиропотока |
| **Устойчивость навигации** | Глушение/отражения сигнала в «каньонах», тоннелях | Потеря событий прибытия/отправления, сбои в ETA |
| **Инфраструктурные затраты** | Активные системы (камеры, шлагбаумы) — от 500 тыс. ₽/остановку | Низкая масштабируемость, высокий CAPEX для регионов |
| **Безопасность данных** | Передача статических ID в эфир — риск спуфинга | Уязвимость транспортных систем к манипуляциям |

### 🔹 Решение
**Платформа доверенного локального позиционирования** на базе существующей сети iBeacon:
- ✅ Использует уже установленные метки на остановках Тверской области (100% готовность инфраструктуры)
- ✅ Не передаёт статические идентификаторы в эфир — дешифровка только в защищённом контуре
- ✅ Работает офлайн в мобильном приложении (см. `t1_ble_scanner`) — не требует постоянного подключения
- ✅ Масштабируется на другие дорожные объекты без смены архитектуры

| Показатель | Значение | Метод оценки |
