# Billing Architecture

Биллинг получает usage-события от основного сервиса, агрегирует их по `partner` и `client_id`, рассчитывает квоты, overage и формирует данные для счетов и вебхуков.

## Компоненты

- `Usage Aggregator` — собирает и агрегирует события использования;
- `Billing Engine` — рассчитывает лимиты, перерасход и стоимость;
- `Partner Billing API` — предоставляет отчётность и операции управления `client_id`;
- `Webhook Dispatcher` — отправляет уведомления о достижении квот и выпуске счетов.

## Поток данных

Каждый успешный вызов основного API формирует usage-событие. После агрегации оно попадает в расчётный контур, где становится доступным для квотирования, отчётности и коммерческого биллинга.

## Операционные требования

- данные usage должны быть доступны в отчётах не позднее следующего расчётного окна;
- ключевые операции биллинга должны быть идемпотентны;
- журналы биллинговых изменений должны быть доступны для аудита;
- расчёт overage должен быть воспроизводимым по сохранённым usage-данным.
