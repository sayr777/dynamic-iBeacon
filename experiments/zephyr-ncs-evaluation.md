# Эксперимент: переход на Zephyr / nRF Connect SDK

## Цель

Рассматривать `Zephyr` не как немедленную замену текущей прошивки, а как **отдельную ветку / эксперимент**.
Подход оправдан, если нужно проверить:

- насколько удобно будет сопровождать BLE-метку в экосистеме `Zephyr`;
- можно ли сохранить режим `single advertising event + deep sleep`;
- не ухудшится ли средний ток и сложность производства;
- даст ли `nRF Connect SDK` преимущества для будущих функций.

## Текущая база проекта

На момент подготовки этой заметки основной проект уже использует не `Arduino SDK`, а:

- `nRF5 SDK 17.1.x`;
- `SoftDevice S112`;
- `nRF52832`;
- собственную платформенную реализацию для:
  - iBeacon advertising;
  - смены `Major/Minor/MAC`;
  - RTC wakeup;
  - `System OFF`;
  - защиты через `APPROTECT`.

То есть переход на `Zephyr` для данного репозитория означает не "уйти с Arduino", а **переписать платформенный слой под другой стек, другую сборку и другую модель конфигурации**.

## Предварительная оценка сложности

### Изучение Zephyr

- собрать и прошить базовый sample: `2-5 дней`;
- освоить `west`, `CMake`, `Kconfig`, `prj.conf`, `devicetree`, flashing/debug: `1-2 недели`;
- уверенно доводить low-power BLE-устройство до измеримого результата: `2-6 недель`.

### Перенос именно этой BLE-метки

- черновой proof-of-concept для опытного embedded-разработчика: `3-7 рабочих дней`;
- аккуратный перенос с повторной проверкой сна, пробуждения, advertising и потребления: `1-2 недели`;
- если `Zephyr` новый для команды: чаще реалистично `2-4 недели`.

## Почему это не быстрый SDK-swap

В текущем проекте критична не только BLE-функциональность, но и поведение на границе энергопотребления:

- один advertising event вместо длительной рекламы;
- смена random static MAC на каждый слот;
- пробуждение по RTC;
- немедленный возврат в глубокий сон;
- предсказуемое поведение после cold boot из `System OFF`.

На `Zephyr` всё это реализуемо, но потребует заново проверить:

- BLE API и способ обновления advertising payload;
- управление адресом устройства;
- low-power path;
- сохранение состояния между пробуждениями;
- фактический ток на железе, а не только успешную сборку.

## Когда эксперимент с Zephyr оправдан

Имеет смысл делать отдельную ветку, если нужен хотя бы один из пунктов:

- единый современный стек для нескольких Nordic-проектов;
- перспектива расширения устройства: сенсоры, OTA, больше BLE-логики;
- желание перейти на стандартную экосистему `west` и `nRF Connect SDK`;
- подготовка к будущим платформам Nordic, где `NCS/Zephyr` уже основной путь.

## Когда лучше оставить текущий стек

Для узкой задачи "BLE-маяк с очень низким потреблением" текущая база на `nRF5 SDK + S112` остаётся сильным вариантом, если приоритеты такие:

- минимальный объём прошивки;
- меньше абстракций;
- быстрое доведение до производства;
- минимум миграционных рисков;
- проверенное поведение в `System OFF`.

## Риски эксперимента

- можно получить рабочий beacon, но потерять преимущество по потреблению;
- больше инфраструктуры сборки и зависимостей по сравнению с `nRF5 SDK`;
- часть времени уйдёт не на прикладную логику, а на освоение `west/Kconfig/devicetree`;
- потребуется отдельный цикл измерений тока, потому что "собралось" не означает "годится для батарейной метки".

## Минимальный план эксперимента

### Этап 1. Базовая среда

- поднять отдельный workspace `nRF Connect SDK`;
- собрать и прошить стандартный `iBeacon` sample;
- проверить flashing/debug на используемом программаторе.

### Этап 2. Поведение маяка

- заменить фиксированные `UUID/Major/Minor` на проектную схему;
- реализовать смену `Major/Minor/MAC` по слоту;
- проверить, что реклама действительно идёт в требуемом режиме.

### Этап 3. Low-power

- поднять wakeup по RTC;
- реализовать сценарий, эквивалентный текущему `System OFF`;
- измерить ток сна и средний ток цикла.

### Этап 4. Сравнение с текущей реализацией

- сравнить Flash/RAM;
- сравнить сложность сборки и прошивки;
- сравнить ток сна и средний ток;
- принять решение: продолжать порт или оставить эксперимент как исследование.

## Предварительная рекомендация

Для этого проекта `Zephyr` разумно вести как **исследовательскую ветку**, а не как немедленную замену основной прошивки.

Практический критерий успеха эксперимента:

- сохранён сценарий `advertise once -> sleep`;
- нет критического ухудшения тока;
- сборка и прошивка воспроизводимы;
- поддержка проекта в новой экосистеме действительно становится проще.

Если хотя бы один из этих пунктов не выполняется, текущий стек на `nRF5 SDK` остаётся предпочтительным.
