# Алгоритм идентификации меток

## Цель

Обеспечить:
- единый статичный `iBeacon UUID` для всех меток нашего оператора;
- динамические `Major` и `Minor`, не раскрывающие `TagID` из эфира;
- локальное восстановление `TagID` на нашем сервере;
- сосуществование в одном регионе с метками других операторов.

## Параметры

| Параметр | Размер | Описание |
|---|---|---|
| `OPERATOR_UUID` | 16 байт | статичный `iBeacon UUID` нашего оператора |
| `TAG_ID` | 16 бит | внутренний идентификатор метки |
| `KEY` | 16 байт | секретный ключ региона |
| `SLOT_DURATION` | 300 с | длительность временного слота |
| `slot` | 32 бита | `unix_time // SLOT_DURATION` |

## Генерация Major / Minor

На каждый слот выполняется один вызов `AES-128 ECB`.

```text
block[0]  = TAG_ID >> 8
block[1]  = TAG_ID & 0xFF
block[2]  = slot >> 24
block[3]  = slot >> 16
block[4]  = slot >> 8
block[5]  = slot & 0xFF
block[6..15] = 0x00

out   = AES-128-ECB(KEY, block)
major = (out[0] << 8) | out[1]
minor = (out[2] << 8) | out[3]
mac   = out[4..9]
mac[0] |= 0xC0
```

## Что в эфире

| Поле | Статус |
|---|---|
| `UUID` | статичный, общий для оператора |
| `Major` | меняется каждый слот |
| `Minor` | меняется каждый слот |
| `RadioMAC` | меняется через BLE Privacy |
| `TAG_ID` | в эфир не передаётся |

## Серверная логика

Сканер или БНСО передают:

```text
UUID, Major, Minor, RSSI, timestamp
```

Серверная обработка:

```text
1. Сравнить UUID со справочником операторов.
2. Если UUID принадлежит нам:
   - выполнить локальный lookup по Major/Minor
   - восстановить TagID
3. Если UUID принадлежит внешнему оператору:
   - вызвать внешний REST оператора
   - получить TagID и метаданные
4. Если UUID неизвестен:
   - пометить пакет как unknown operator
```

## Локальный lookup для нашего оператора

```python
slot_now = unix_time // SLOT_DURATION

for slot in [slot_now - 1, slot_now, slot_now + 1]:
    for tag_id in range(0, NUM_TAGS + 1):
        block = pack(tag_id, slot)
        out = AES128(KEY, block)
        major_c = (out[0] << 8) | out[1]
        minor_c = (out[2] << 8) | out[3]
        if major_c == major_rx and minor_c == minor_rx:
            return tag_id

return None
```

Окно `slot ± 1` нужно для компенсации дрейфа часов и задержек доставки.

## Реестр операторов

Справочник операторов хранит:
- имя оператора;
- статичный `UUID`;
- способ разрешения:
  - локально;
  - через внешний `REST`.

Пример: [server/operators.example.json](../server/operators.example.json)

## Внешний REST-маршрут

Для чужого оператора RNIS-сервер передаёт минимум:

```json
{
  "major": 43143,
  "minor": 40635,
  "rssi": -71
}
```

Ответ ожидается в формате оператора и содержит как минимум его `TagID`.

## Свойства безопасности

- Статичный `UUID` определяет только оператора, а не конкретную метку.
- Без `KEY` наблюдатель не может восстановить `TagID` по `Major/Minor`.
- Подбор `KEY` сводится к атаке на `AES-128`.
- Даже зная `UUID`, внешний наблюдатель получает только принадлежность к оператору, но не уникальный постоянный идентификатор метки.
- `RadioMAC` не используется как идентификатор и может меняться независимо через BLE Privacy.

## Легаси-совместимость

Для существующих БНСО сохранён режим:
- `Умка`: `ID = Major * 65536 + Minor`
- `Скаут`: `ID = Major + Minor`

Он нужен только для переходного периода, пока БНСО не передают `UUID + Major + Minor + RSSI`.

## Пример

```text
UUID   = FDA50693-A4E2-4FB1-AFCF-C6EB07647825
TAG_ID = 42
slot   = 5922532
KEY    = 2B7E151628AED2A6ABF7158809CF4F3C

block  = 00 2A 00 5A 5F A4 00 00 00 00 00 00 00 00 00 00
out    = AES-128-ECB(KEY, block)

Major  = out[0:2]
Minor  = out[2:4]
```
