# Архитектура изделия

## Назначение

Изделие передаёт `iBeacon`-посылки для локального позиционирования дорожных объектов.

Ключевая идея:
- `UUID` указывает только на оператора;
- `Major` и `Minor` идентифицируют метку внутри оператора только для сервера, знающего `KEY`.

## Платформа

Основная production-платформа: [YJ-16013](../specs/YJ-16013-datasheet.pdf) на `nRF52832`.

## Логическая схема

### На устройстве

1. Хранится `TAG_ID`, `KEY`, `OPERATOR_UUID`.
2. При смене слота вычисляются новые `Major` и `Minor`.
3. Передаётся стандартный `iBeacon`:
   - `UUID = OPERATOR_UUID`
   - `Major = dynamic`
   - `Minor = dynamic`

### На сервере

1. Приходит пакет `UUID + Major + Minor + RSSI + timestamp`.
2. `UUID` определяет оператора.
3. Для нашего оператора выполняется локальный lookup.
4. Для внешнего оператора выполняется REST-вызов в его систему.

## Основной цикл метки

```text
RTC wakeup -> check slot -> maybe update Major/Minor -> advertise once -> System OFF
```

### Обновление параметров

```text
slot  = unix_time / 300
block = tag_id || slot || 0x00[10]
out   = AES128(KEY, block)

Major = out[0:2]
Minor = out[2:4]
```

## Сетевое взаимодействие

### Наш оператор

```text
BLE/BNSO -> RNIS -> local lookup -> TagID -> stop/object registry
```

### Внешний оператор

```text
BLE/BNSO -> RNIS -> detect foreign UUID -> external REST -> foreign TagID
```

## Потребление

| Фаза | Средний вклад |
|---|---:|
| Deep sleep + RTC | ~1.5 µА |
| TX событие | ~2.65 µА |
| AES и обновление параметров | ~0.07 µА |
| LDO | ~1.0 µА |
| **Итого** | **~5.2 µА** |

## Почему статичный UUID лучше для интеграции

- RNIS сразу определяет оператора без предварительного AES-поиска.
- Можно обслуживать несколько операторов в одном регионе.
- Упрощается интеграция с БНСО и внешними REST-сервисами.
- Внешнему наблюдателю всё равно не раскрывается `TagID`.
